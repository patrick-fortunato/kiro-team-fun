# Work Summary Maintenance

Whenever you make meaningful changes to this project (code edits, new files, config changes, git operations, architectural decisions), update `WORK-SUMMARY.md` in the project root to reflect the current state.

## What to update

- **"Current State" date** — update to today's date
- **"What's Done"** — add or modify entries if tasks are completed or new work is started
- **"Git State"** — update branch info, PR status, or commit references if they change
- **"Key Files"** — add new files if they become important, remove if deleted
- **"Environment"** — update if dependencies, env vars, or setup steps change
- **"What's Next"** — move items out as they're completed, add new ideas or decisions

## When to update

- After completing a feature or fix
- After git operations (commits, merges, branch changes, PRs)
- After adding/removing significant files or dependencies
- After architectural decisions or design changes
- At the end of a working session if changes were made

## How to update

- Keep it concise and scannable — this is a quick-reference doc, not a changelog
- Use the existing format and table structure
- Don't duplicate what's already in the docs/ folder — just reference it
- Focus on "where are we now" and "what's next" over detailed history
